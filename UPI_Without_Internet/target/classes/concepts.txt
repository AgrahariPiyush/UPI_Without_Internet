Drawbacks

2) No regulatory compliance layer : Real UPI: Every transaction goes through NPCI's switch, is logged in a regulated ledger, is subject to RBI guidelines on transaction limits (UPI Lite is capped at ₹500 per transaction, ₹2000 total wallet balance),
 and every bridge node equivalent is a licensed Payment Service Provider.

3)H2 in-memory database means zero durability :
-Real payment system: PostgreSQL with synchronous replication — a transaction is only confirmed after it's written to at least one replica.
The ledger survives server crashes.
- H2 in-memory means the entire ledger disappears on server restart. If the Spring Boot process crashes mid-settlement (after debit, before ledger write), the data is gone. @Transactional handles application-level
 atomicity but can't protect against process death if the DB itself is volatile.

-background BLE scanning is rate-limited, apps can't maintain GATT connections while backgrounded, and iOS peripheral mode is locked down almost entirely for third-party apps.
-So we directly use simulating all five "phones" as in-memory objects on one JVM.

5)Single JVM idempotency check :

Invariants

1)RSA can only encrypt ~245 bytes for a 2048-bit key. A PaymentInstruction JSON is ~300 bytes.
RSA is also slow.AES-GCM is ~100x faster and handles unlimited data. So RSA is used only to encrypt the 32-byte AES key (key wrapping), and AES does the actual payload.

2)Use of IV and should be unique per packet :

3)Hash Idempotency insread of pakedID
-The ciphertext is cryptographically authenticated by GCM, so any modification is detectable.
-Two legitimate deliveries of the same payment are byte-for-byte identical →
same SHA-256 hash → idempotency cache catches the duplicate.


4)RSA vs GCM
-RSA provides confidentiality of the AES key — only the server can unwrap it.
-GCM provides both confidentiality (encryption) AND integrity (authentication tag) of the payload.
